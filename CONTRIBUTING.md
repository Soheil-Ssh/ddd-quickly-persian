# راهنمای مشارکت

از اینکه برای بهبود ترجمه فارسی کتاب *Domain Driven Design Quickly* وقت می‌گذارید، ممنونم.

هدف این راهنما این است که گزارش خطاها و تغییرات پیشنهادی، اطلاعات کافی داشته باشند و بررسی آن‌ها سریع‌تر و دقیق‌تر انجام شود.

## پیش از مشارکت

لطفاً پیش از ایجاد `Issue` یا ارسال `Pull Request`، این فایل‌ها را مطالعه کنید:

- [README](./README.md)
- [پیشگفتار مترجم](./Book/00-03-Translator-Preface.md)
- [مجوز و حقوق اثر](./LICENSE.md)

همچنین بررسی کنید که برای موضوع موردنظر، `Issue` یا `Pull Request` مشابهی از قبل ثبت نشده باشد.

این ترجمه در کنار پروژه [ترجمه فارسی کتاب اصلی اریک اوانز](https://github.com/Soheil-Ssh/ddd-tackling-complexity-persian) توسعه داده می‌شود. هنگام پیشنهاد تغییر برای اصطلاحات کلیدی `DDD`، کاربرد آن اصطلاح در مخزن دیگر را نیز بررسی کنید تا دو پروژه تا حد امکان هماهنگ باقی بمانند.

## چگونه می‌توانید کمک کنید؟

مشارکت‌ها می‌توانند شامل موارد زیر باشند:

- گزارش ترجمه نادرست، ناقص یا نامفهوم
- پیشنهاد معادل دقیق‌تر برای اصطلاحات فنی
- بررسی هماهنگی اصطلاحات با ترجمه کتاب اریک اوانز
- اصلاح خطاهای املایی، نگارشی و نشانه‌گذاری
- اصلاح قالب‌بندی فایل‌های `Markdown`
- گزارش یا اصلاح لینک‌ها و مسیرهای تصاویر
- بررسی فنی مفاهیم `Domain-Driven Design`
- بهبود خوانایی متن، بدون تغییر معنای متن اصلی

## انتخاب میان Issue و Pull Request

### Issue ایجاد کنید، اگر:

- درباره درست‌بودن یک ترجمه مطمئن نیستید.
- پیشنهاد شما روی چند فصل یا تعداد زیادی فایل اثر می‌گذارد.
- قصد دارید معادل یک اصطلاح کلیدی را تغییر دهید.
- تغییر پیشنهادی ممکن است روی ترجمه کتاب اریک اوانز نیز اثر بگذارد.
- برای اعمال تغییر به گفت‌وگو یا تصمیم مشترک نیاز است.
- مشکل را شناسایی کرده‌اید، اما فرصت اصلاح مستقیم آن را ندارید.

از [فرم‌های ثبت Issue](https://github.com/Soheil-Ssh/ddd-quickly-persian/issues/new/choose) استفاده کنید تا اطلاعات لازم به‌صورت منظم ثبت شوند.

### Pull Request ارسال کنید، اگر:

- تغییر کوچک، روشن و قابل‌بررسی است.
- یک خطای املایی یا نگارشی مشخص را اصلاح می‌کنید.
- یک لینک یا مسیر تصویر خراب را اصلاح می‌کنید.
- ترجمه نادرست را با دلیل روشن و بدون ایجاد تغییر گسترده اصلاح می‌کنید.

برای تغییرات گسترده، ابتدا یک `Issue` ایجاد کنید و پس از توافق درباره مسیر اصلاح، `Pull Request` بفرستید.

## اصول ترجمه

در این پروژه، هدف ترجمه تحت‌اللفظی نیست. ترجمه باید معنای متن اصلی را دقیق منتقل کند و در عین حال برای خواننده فارسی‌زبان طبیعی و قابل‌فهم باشد.

هنگام پیشنهاد یا اعمال تغییر، این اصول را در نظر بگیرید:

1. **معنا بر شباهت ظاهری جمله اولویت دارد.** ساختار جمله فارسی لازم نیست دقیقاً از ترتیب واژه‌های متن انگلیسی پیروی کند.
2. **مفهوم فنی نباید ساده‌سازی یا تغییر داده شود.** کوتاه و مقدماتی‌بودن کتاب نباید باعث شود قیدها، تفاوت‌ها یا جزئیات معنایی حذف شوند.
3. **توضیح جدید به متن نویسندگان اضافه نکنید.** اگر بخشی نیازمند توضیح مترجم است، ابتدا درباره آن `Issue` ایجاد کنید.
4. **اصطلاحات کلیدی باید یکدست باشند.** پیش از تغییر یک معادل، کاربرد آن را در این مخزن و مخزن ترجمه کتاب اریک اوانز بررسی کنید.
5. **نام عناصر برنامه‌نویسی ترجمه نمی‌شود.** نام کلاس‌ها، متدها، متغیرها، `Namespace`ها و قطعه‌کدها باید بدون تغییر باقی بمانند.
6. **ارتباط با منابع انگلیسی حفظ شود.** اصطلاحاتی که در جامعه نرم‌افزار با نام انگلیسی شناخته می‌شوند، مطابق سیاست توضیح‌داده‌شده در پیشگفتار نوشته می‌شوند.
7. **لحن فشرده کتاب حفظ شود.** از طولانی‌کردن غیرضروری جمله‌ها و تبدیل متن خلاصه به توضیحی بسیار مفصل خودداری کنید.

## هماهنگی اصطلاحات میان دو پروژه

این مخزن و مخزن `ddd-tackling-complexity-persian` باید تا حد ممکن از معادل‌های هماهنگ استفاده کنند.

پیش از پیشنهاد تغییر یک اصطلاح کلیدی:

- اصطلاح را در هر دو مخزن جست‌وجو کنید.
- مشخص کنید معادل فعلی در هر پروژه چیست.
- توضیح دهید آیا پیشنهاد فقط برای این کتاب مناسب است یا باید در هر دو پروژه بررسی شود.
- در صورت اثرگذاری گسترده، برای هر مخزن `Issue` مستقل ایجاد کنید و میان آن‌ها لینک قرار دهید.

هماهنگی لزوماً به معنای استفاده اجباری از یک ترجمه در تمام بافت‌ها نیست. اگر مفهوم یا کاربرد یک واژه در دو متن متفاوت باشد، این تفاوت باید همراه با دلیل روشن توضیح داده شود.

## شیوه نگارش و قالب‌بندی

برای یکدست‌ماندن متن، این قواعد را رعایت کنید:

- متن فارسی باید رسمی، روان و طبیعی باشد؛ نه محاوره‌ای و نه بیش از حد ادبی.
- نیم‌فاصله در واژه‌هایی مانند «می‌شود»، «به‌صورت»، «قابل‌فهم» و «فارسی‌زبان» رعایت شود.
- از علائم نگارشی فارسی استفاده شود.
- اصطلاحات فنی انگلیسی، نام الگوها و عناصر کد با قالب کد درون‌خطی نوشته شوند؛ مانند `Aggregate` و `Bounded Context`.
- عنوان کتاب‌ها با قالب *Italic* نوشته شود.
- متن داخل بلوک‌های کد تغییر یا ترجمه نشود، مگر آنکه خود کد دارای خطای قطعی باشد.
- ساختار عنوان‌ها، فهرست‌ها، تأکیدها، تصاویر و ارجاعات متن اصلی بدون دلیل تغییر نکند.
- لینک‌های ناوبری ابتدا و انتهای فایل‌ها پس از تغییر نام یا جابه‌جایی بررسی شوند.
- از تغییرات سراسری قالب‌بندی در کنار اصلاح ترجمه خودداری شود. هر `Pull Request` بهتر است فقط یک هدف مشخص داشته باشد.

## پیشنهاد یا تغییر اصطلاحات فنی

تغییر معادل یک اصطلاح ممکن است روی چند فصل و حتی هر دو مخزن ترجمه اثر بگذارد. برای چنین تغییراتی ابتدا از قالب «پیشنهاد اصطلاح فنی» یک `Issue` ایجاد کنید و این موارد را توضیح دهید:

- اصطلاح انگلیسی
- معادل فعلی در این مخزن
- معادل فعلی در مخزن کتاب اریک اوانز، در صورت وجود
- معادل پیشنهادی
- بخش یا فصل مرتبط
- دلیل انتخاب معادل جدید
- نمونه کاربرد در متن
- دامنه تغییر پیشنهادی
- منبع یا استدلال فنی، در صورت وجود

پیشنهاد یک معادل لزوماً به‌معنای جایگزینی کامل آن در تمام کتاب نیست. تصمیم نهایی با توجه به معنا، بافت، رایج‌بودن اصطلاح، خوانایی متن و امکان جست‌وجو در منابع انگلیسی گرفته می‌شود.

## ارسال Pull Request

### ۱. مخزن را Fork و Clone کنید

```bash
git clone https://github.com/<your-username>/ddd-quickly-persian.git
cd ddd-quickly-persian
```

### ۲. یک Branch مستقل بسازید

نام Branch باید کوتاه و مرتبط با تغییر باشد. برای مثال:

```bash
git checkout -b fix/chapter-02-translation
git checkout -b docs/chapter-01-typo
git checkout -b terminology/ubiquitous-language
```

### ۳. تغییر را محدود و قابل‌بررسی نگه دارید

- یک موضوع اصلی را در هر `Pull Request` دنبال کنید.
- از بازنویسی بخش‌های نامرتبط خودداری کنید.
- فایل را در نمای `Preview` گیت‌هاب یا یک نمایشگر `Markdown` بررسی کنید.
- مطمئن شوید لینک‌ها، تصاویر و قالب‌بندی متن خراب نشده‌اند.

### ۴. Commit روشن بنویسید

پیام Commit بهتر است کوتاه و مشخص باشد. برای مثال:

```text
fix: correct Aggregate translation in chapter 3
docs: fix Persian punctuation in chapter 1
fix: repair next chapter navigation link
```

### ۵. Pull Request را کامل توضیح دهید

در توضیح `Pull Request` مشخص کنید:

- چه چیزی تغییر کرده است؟
- چرا این تغییر لازم بوده است؟
- کدام فایل‌ها یا بخش‌ها تحت‌تأثیر قرار گرفته‌اند؟
- آیا اصطلاح فنی جدید یا معادل متفاوتی پیشنهاد شده است؟
- آیا تغییر با مخزن ترجمه کتاب اریک اوانز مرتبط است؟
- آیا `Issue` مرتبطی وجود دارد؟

## چک‌لیست پیش از ارسال Pull Request

- [ ] تغییر فقط به یک موضوع مشخص مربوط است.
- [ ] متن فارسی با متن اصلی مقایسه شده است.
- [ ] معنای فنی متن تغییر نکرده است.
- [ ] اصطلاحات با بخش‌های دیگر این کتاب هماهنگ هستند.
- [ ] اصطلاحات کلیدی با مخزن ترجمه کتاب اریک اوانز بررسی شده‌اند.
- [ ] نیم‌فاصله و علائم نگارشی بررسی شده‌اند.
- [ ] قالب‌بندی `Markdown` سالم است.
- [ ] لینک‌ها و مسیر تصاویر بررسی شده‌اند.
- [ ] هیچ فایل اسکن‌شده، نسخه کامل کتاب یا محتوای بدون مجوز اضافه نشده است.
- [ ] توضیح کافی برای بررسی تغییر در `Pull Request` نوشته شده است.

## حقوق اثر و محتوای مجاز

برای گزارش یا توضیح یک مشکل، فقط کوتاه‌ترین بخش لازم از متن اصلی را نقل کنید. از قراردادن صفحه کامل، فصل کامل، تصویر اسکن‌شده، فایل کتاب یا بخش‌های طولانی متن اصلی در `Issue` یا `Pull Request` خودداری کنید.

نسخه انگلیسی کتاب در InfoQ به‌صورت رایگان قابل دریافت است، اما رایگان‌بودن دسترسی به آن به‌معنای آزادبودن ترجمه، بازنشر یا توزیع مجدد محتوا نیست.

موارد زیر نباید به مخزن اضافه شوند:

- نسخه `PDF`، `EPUB`، `MOBI` یا اسکن کتاب اصلی
- ترجمه‌های متعلق به اشخاص یا ناشران دیگر
- تصاویر یا نمودارهایی که اجازه استفاده از آن‌ها مشخص نیست
- محتوای شخص ثالث بدون منبع یا مجوز روشن

با ارسال مشارکت، تأیید می‌کنید که حق ارائه آن را دارید و شرایط بخش «Contributions» در [LICENSE.md](./LICENSE.md) را می‌پذیرید. مشارکت در این مخزن هیچ حقی نسبت به اثر اصلی یا محتوای متعلق به اشخاص ثالث ایجاد نمی‌کند.

## بررسی و پذیرش تغییرات

هر مشارکت بر اساس این موارد بررسی می‌شود:

- دقت نسبت به متن اصلی
- درستی مفهوم فنی
- هماهنگی با اصطلاحات موجود در هر دو پروژه
- روانی و طبیعی‌بودن نثر فارسی
- حفظ لحن کوتاه و فشرده کتاب
- محدود و قابل‌بررسی‌بودن تغییر
- رعایت قالب‌بندی و ساختار مخزن
- رعایت حقوق اثر و شرایط مخزن

ممکن است پیش از ادغام، تغییر یا توضیح بیشتری درخواست شود. ارسال `Pull Request` به‌معنای پذیرش قطعی آن نیست، اما تمام پیشنهادهای مستدل با دقت بررسی خواهند شد.

گفت‌وگوها باید محترمانه، مشخص و متمرکز بر متن و استدلال فنی باشند. اختلاف‌نظر درباره ترجمه اصطلاحات طبیعی است؛ بهتر است پیشنهادها با مثال، بافت جمله و دلیل روشن مطرح شوند.

از مشارکت شما در دقیق‌تر و خواناترشدن این ترجمه ممنونم.
